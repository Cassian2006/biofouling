# 第一批下载范围建议

## 你现在要下载哪些数据

当前优先级应该按下面顺序走。

### 1. AIS 区域发现数据

用途：

- 在固定 bbox 内先找出这段时间里有哪些船出现过
- 生成首批 MMSI 列表
- 判断研究区在这个时间窗内的船舶密度和数据量级

为什么先下它：

- 这是后续 MMSI 轨迹下载的前置步骤
- 不先做区域发现，就不知道后面是抓几十艘还是几千艘

### 2. AIS 轨迹数据

用途：

- 对筛出来的 MMSI 拉取时间切片轨迹
- 用于清洗、特征工程、FPI/ECP 计算和单船报告

建议字段：

- `mmsi`
- `timestamp`
- `latitude`
- `longitude`
- `sog`
- `cog`
- `heading`
- `nav_status`
- `ship_type`

### 3. 环境数据

第一阶段先下：

- `sst`
- `chlorophyll_a`

第二阶段再补：

- `salinity`
- `current_u`
- `current_v`

## 时间区间怎么选

你现在 bbox 已经固定，时间区间还没定。不要一上来就下很长时间，建议分三层。

### 方案 A：冒烟测试

- 时长：`3 天`
- 目的：验证 API 可用性、切片策略、字段完整性、磁盘落盘和清洗流程
- 适用：第一次正式下载前

### 方案 B：首批试运行

- 时长：`14 天`
- 目的：验证项目主链路是否能稳定生成特征和报告
- 适用：第一批真正用于开发的数据

### 方案 C：第一阶段正式样本

- 时长：`30 天`
- 目的：形成有代表性的首版分析样本
- 适用：当 14 天流程已经稳定后

## 当前推荐

如果现在立刻开始，我建议：

1. 先做 `3 天` 区域发现
2. 再做 `7-14 天` 的首批 MMSI 轨迹下载
3. 流程稳定后再扩到 `30 天`

## 为什么不建议直接下一个月甚至更久

- 你还没验证真实 API 返回量
- 还没验证切片大小是否合适
- 还没验证单个时间窗内会命中多少 MMSI
- 一旦请求过大，失败重试会把时间浪费在错误配置上

## 切片建议

### 区域发现

- 先按 `6 小时` 切片
- 如果返回量仍大，再降到 `3 小时`

### 轨迹下载

- 先按 `6 小时` 切片
- 如果某些船数据特别密，再按 `1-3 小时` 切

## 当前下载顺序

1. 选一个 `3 天` 窗口做区域发现
2. 从区域结果提取 MMSI
3. 选一批 MMSI 做 `7-14 天` 轨迹下载
4. 同步准备同时间窗的 `sst` 和 `chlorophyll_a`
5. 进入当前项目的数据清洗与评分流程
